Why Not Just a Website?

I tried web timers first. Here's what happened:

  • Closed by accident: One wrong click and my timer is gone
  • No real notifications: Browser tabs can't interrupt me properly
  • Only works in browser: When browser is closed, no reminders
  • Data scattered: Different devices, different history

I needed a real desktop app

The Big Question

How do I build a desktop app?

I'm a web developer. I know HTML, CSS, JavaScript.

But desktop apps? That's a different world...

Two Paths Forward

I found two technologies that let web developers build desktop apps:

Electron

The popular choice

Used by VS Code, Slack, Discord

Tauri

The new kid

Built with Rust

Let's Look at the Numbers

Metric Electron Tauri
App Size 120-150 MB 3-10 MB
Memory Used 150-300 MB 50-100 MB
Startup Time 2-3 seconds Under 1 second
Backend Node.js Rust

15x smaller, 3x less memory, 3x faster startup

Why This Matters for StandClock

Remember: this app needs to run all day, every day

  • Memory matters: Less memory = computer stays fast
  • Size matters: Quick to download and update
  • Speed matters: Starts when computer boots
  • Battery matters: For laptop users working remotely

For a background app, Tauri is perfect

So What Exactly Is Tauri?

Think of it as a bridge between web and desktop

  • Frontend: HTML, CSS, JavaScript (any framework!)
  • Backend: Rust (handles system stuff)
  • Smart approach: Uses your OS's browser engine
  • Result: Small, fast, secure apps

Web UI + Rust backend + System browser = Tiny desktop app

Time to Build

Here's what I wanted in the app:

  • Simple timer interface
  • Choose break activities (stretch, walk, water)
  • Track my break history
  • Customizable (I like dark themes)
  • Notifications when time is up

Sounds simple, right? Well...

Attempt 1: Go Full Rust

My ambitious plan: Use Rust for everything

  • Frontend: Sycamore (Rust framework with WASM)
  • Backend: Rust (obviously)
  • Goal: Pure Rust, no JavaScript

This failed spectacularly

Why It Failed

  • Sycamore was too new: Almost no documentation or examples
  • Everything took forever: Simple UI changes = hours of work
  • No component libraries: Had to build everything from scratch
  • Debugging was painful: Error messages were cryptic

After 2 weeks, I had a button and a timer. That's it.

Time to pivot

Attempt 2: Hybrid Approach

New plan: Use the right tool for each job

  • Frontend: Solid.js (familiar, fast, good docs)
  • Backend: Rust (for system operations)
  • UI Library: DaisyUI (32 themes for free!)
  • Database: SQLite (simple, local)

This worked beautifully

How It All Fits Together

Frontend (Solid.js):

  • Timer UI, buttons, settings
  • Talks to Rust backend via Tauri commands

Backend (Rust):

  • Manages timer logic
  • Saves data to SQLite
  • Triggers notifications
  • Handles app startup/shutdown

Tauri (The Bridge):

  • Connects frontend to backend safely
  • Manages the window and system tray

Why Rust in the Backend Works Great

  • Type safety: Catches bugs at compile time
  • No garbage collector: Predictable performance
  • Fast database operations: SQLite queries are instant
  • Memory efficient: Perfect for background apps
  • Safe concurrency: Timer runs smoothly without blocking

Rust handles the heavy lifting, JavaScript handles the pretty UI

The Hard Parts

Learning Rust while building was tough

  • Ownership & borrowing: Rust's core concepts took time
  • Compiler errors: Helpful but overwhelming at first
  • Async programming: Different from JavaScript promises
  • Less Stack Overflow: Fewer answers for Tauri-specific issues
  • Platform quirks: Windows, Mac, Linux all behave slightly differently

But I got through it, and you can too

What Went Really Well

  • Performance: App uses only 50MB RAM, starts instantly
  • Final app size: 6MB download (compared to 150MB for Electron)
  • Development speed: Once past Rust learning curve, very productive
  • Type safety: TypeScript + Rust caught bugs before testing
  • Clean architecture: Clear separation between UI and logic
  • DaisyUI themes: 32 beautiful themes with minimal effort

Key Lessons

  • Don't go full Rust on frontend: Use mature JS frameworks
  • Start simple: Get basic version working before adding features
  • Rust is worth learning: But budget time for the learning curve
  • Community matters: Join Tauri Discord for help
  • Test on all platforms early: Don't wait until the end

Should You Use Tauri?

It depends on your project

Tauri Is Great For:

  • Background apps: Tools that run all day
  • Utility apps: Small, focused tools
  • Performance-critical apps: When speed matters
  • New projects: Starting fresh is easier
  • Learning opportunity: Want to learn Rust

Examples: timers, system monitors, small productivity tools

Stick with Electron If:

  • Team velocity matters: Need to ship fast
  • JavaScript-only team: No time to learn Rust
  • Complex apps: Need lots of Node.js libraries
  • Existing project: Already built with Electron
  • Identical everywhere: Exact same look on all platforms

Examples: IDEs, design tools, large enterprise apps

My Verdict

Tauri was the right choice for StandClock

  • App runs silently in background all day
  • Users don't even notice it's there (performance-wise)
  • 6MB download means fast updates
  • Learning Rust opened new opportunities

Was it harder? Yes. Was it worth it? Absolutely.

Key Takeaways

  • Match the tool to the problem: Tauri for lightweight, Electron for complex
  • Rust has a learning curve: But it pays off for performance apps
  • Don't reinvent the wheel: Use mature JS frameworks for UI
  • Start simple, iterate: Get it working before making it perfect
  • Community matters: Join Discord, ask questions, share learnings
